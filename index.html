The System · Two Independent Projects

Two separate projects, one recruiting funnel

This platform is two completely independent projects, each its own codebase, repository, and deployment. They talk only through the shared Notion workspace, so either can be changed or redeployed without touching the other.

Project 01 · independent codebase

◆ Role Matching

Python · FastAPI · LLM scoring

A webhook-driven matching engine that turns a free-text job description into a structured rubric, then scores and ranks the candidate pool through a six-stage funnel, upward of 20,000 LLM API calls per run depending on the role. Impact

What it delivers, and how it works

The whole point: manual processes that used to take weeks now finish in seconds, minutes, or at most a couple of hours. Two audiences read a system like this differently, so here is both the outcome and the workflow that produces it.

For the business

  • A single 90 to 120 minute run on Railway scans the full 40,000-candidate database and scores upward of 20,000 of them, returning a ranked, explained shortlist for a role, work that is otherwise weeks of manual screening.
  • Completed interviews are processed automatically: the transcript is fetched in full and the candidate's record and status are updated in Notion within about 60 seconds, with no recruiter touching the card.
  • Communications run at scale every day: 2,000+ WhatsApp messages and emails, with CV parsing done in under a minute.
  • The team stays informed automatically: each matching run posts Slack status updates at start, progress, and completion, and every match ships with a written reason, so the team can act on the shortlist on sight.

For the technical team

  • AI only where it earns its keep: DeepSeek is reserved for the judgment-heavy steps; parsing, filtering, and routing are handled by fast deterministic logic and regex, so tokens are never wasted.
  • Event-driven: a workspace change fires a webhook, the right Railway service handles it, the result is written back.
  • A run makes 20,000+ DeepSeek API calls, kept inside quota by a sequential queue, concurrency caps, and backoff; the cheap-filter-first funnel narrows the pool before any paid call.
  • Transcript fetches retry for up to 10 minutes until the full transcript is guaranteed, so scoring never runs on a partial interview.
  • Idempotent re-runs, per-candidate error isolation, and schema-drift defense keep an unattended run from corrupting data.

Architecture

How data moves through the platform

Most work is event-driven off the recruiting team's workspace: a record changes, a webhook fires, the right service picks it up, does one job well, and writes the result back. Time-based reminder sweeps run on a cron schedule instead. No shared database to corrupt, no monolith to coordinate.

Every box is an isolated, independently deployable service. Failure in one (e.g. WhatsApp) cannot block another (e.g. matching). The workspace is the single source of truth; services are stateless and idempotent.

Deep Dive · The Matching Engine

Six stages from job description to ranked shortlist

The hard problem isn't calling an LLM, it's calling it responsibly on a large pool: cheaply, in priority order, with explainable output, and without melting API quotas. Each stage narrows the pool before the next, expensive stage runs.

1

Rubric extraction LLM

A free-text job description becomes a structured rubric: hard gates, core vs. preferred requirements, search keywords, and allowed locations. This turns fuzzy hiring intent into something the rest of the pipeline can filter and score against deterministically.

2

Keyword pool reduction cheap filter

A fast, local keyword-overlap pass drops candidates with no plausible signal before any paid LLM call. This is the single biggest cost-control lever in the system.

3

Hard eligibility filters rules

Non-negotiable gates, language, location, minimum screening score, applied as rules. Missing or unexpected data fields are tolerated and logged, never crashed on (see schema-drift defense).

4

Priority bucketing & ordering heuristics

Survivors are bucketed by business priority (referrals first, then priority backgrounds) and ordered by the best available baseline score. Higher-value candidates are scored first, so a capped run still spends budget on the right people.

5

LLM scoring concurrency-limited · retried

Each candidate gets a 0-100 score and a written rationale from the LLM, under a global concurrency cap. Transport errors, empty completions, and invalid JSON are retried; a single candidate's failure is isolated and skipped rather than killing the run.

6

Qualification & backfill targeting

Candidates above threshold are linked first. If the qualified pool is short of the run's target, the engine backfills with the highest-scoring below-threshold candidates, so the team always gets a usable shortlist, and the fallback is logged transparently.

Under the Hood

The parts that make it production, not a script

Anyone can wire an API to an LLM. Keeping it correct, affordable, and recoverable while it runs unattended against rate-limited third-party services is the actual job.

↻ Idempotent re-runs

Already-scored records are excluded from future runs. The same job can fire twice, from a retry, a duplicate webhook, or a manual re-trigger, without double-contacting candidates or duplicating matches.

⏱ Rate limiting & queueing

Jobs run through a sequential queue with a concurrency cap and inter-write delays, deliberately staying under Notion and LLM quotas instead of hammering them and getting throttled.

⤾ Retry & graceful degradation

Exponential backoff on transient failures; empty-response and invalid-JSON hardening on LLM calls. Individual failures are skipped, the job only fails if zero usable results were produced.

⛨ Schema-drift defense

Third-party schemas change without warning. Missing properties are ignored, unexpected types are logged and skipped, and eligibility is re-checked after caching, so a workspace edit can't silently corrupt a run.

$ AI only where it earns its keep

DeepSeek is reserved for judgment-heavy steps; parsing, filtering, and routing run on regex and deterministic logic. Cheap filters run before any paid call, caps bound spend, and a TTL cache reuses pools, so tokens are never wasted.

⌗ Observability & team comms

Structured logs trace every record by page id, email, stage, and skip reason. Each run also posts Slack status updates at start, progress, and finish, so the whole team sees state without opening a log.

Configurable by Design

One codebase, many deployments, zero forks

Nothing is hard-coded per client. Behavior is driven entirely by environment variables, so the same service is deployed as many times as needed, each instance tuned to a different goal, database, or client, by changing config and redeploying. The code never forks.

Same engine, Flash or Pro

The matching engine runs as two deployments from one codebase: one on DeepSeek Flash for speed and low cost, one on DeepSeek Pro for tougher roles. A single env var picks the model.

One reminder, deployed 3×

The WhatsApp reminder service is deployed three times, each with a different "hours since submission" threshold, so candidates get a staggered sequence of nudges, all from the same code.

Same code, different database

Each branch points at a different Notion database in the same workspace through env vars. New database, new deployment, no code change.

⚙︎ The payoff: when a client wants the same automation pointed at a different database, tuned to a different setting, or run on a different reminder timeframe, it redeploys to a new space in about 5 minutes. New environment variables and a redeploy, never new code.

In Detail

How each automation actually works

The short version is above. Here is what happens step by step, from a candidate applying to a ranked shortlist landing in the team's workspace.

What happens the moment a candidate applies?

A webhook fires the CV-parsing service the instant an application lands. It pulls the raw CV and extracts a structured profile, contact details, experience, skills, and LinkedIn or portfolio links, then writes it straight back to Notion in under a minute. The parsing is done with regex and deterministic logic; AI is only called for the parts that genuinely need reasoning, so no tokens are spent on data a pattern can extract reliably.

How are candidates contacted, and how is that kept under control?

Outreach runs across email (Postmark) and WhatsApp (Twilio). First invites and follow-up reminders are sent two ways: instantly on a webhook, and on a cron schedule, a reminder sweep runs on a timer, checks who is due based on a configurable "hours since submission" threshold, and sends the next nudge.

The platform handles 2,000+ messages a day while staying inside provider caps through hourly and daily rate limits, retries with exponential backoff, cooldowns, and de-duplication, so a candidate is never double-contacted and a provider is never hammered into throttling.

What happens when a screening interview is completed?

The interview platform fires a webhook on completion, but the full transcript is rarely ready that instant. So the service keeps retrying the transcript fetch for up to 10 minutes, polling until the complete transcript is guaranteed, it will not proceed on a partial one. Acting on half a transcript would mean scoring a candidate on half an interview, so this guarantee matters.

Once the full transcript and interview details are confirmed, it writes them to the candidate's Notion record and changes their status automatically, typically within about 60 seconds of the interview finishing. No recruiter has to move the card.

How is an interview actually scored?

Once the complete transcript is in, a separate DeepSeek-powered service scores it against the role, covering fit, consistency, and communication clarity, and writes a structured score plus a written rationale back to Notion. Transport errors, empty completions, and invalid JSON are retried; one bad call is skipped rather than allowed to block the rest.

How does the matching engine scale to a 40,000-candidate database?

A single run scans up to ~40,000 candidate rows and, depending on the role's requirements, scores upward of 20,000 of them, that is 20,000+ DeepSeek API calls in one run. To make that affordable and fast, a six-stage funnel does cheap, deterministic work first: it builds a rubric from the job description, drops no-signal candidates by keyword overlap, applies hard eligibility gates, and orders the rest by priority, so the model only ever sees candidates with real signal.

The full run completes in 90 to 120 minutes on Railway and posts Slack status updates at start, progress, and completion, so the whole team can watch it run and know the moment it finishes.

How do 20,000+ API calls per run not fail or blow the budget?

Calls run through a sequential queue with concurrency caps, inter-write delays, and exponential backoff to stay inside DeepSeek and Notion quotas instead of getting throttled. Each candidate is scored in isolation, so a single failure is skipped, not fatal; the run only fails if zero usable scores come back. Cheap filters and a TTL cache keep the model from ever re-seeing candidates that were already excluded or scored.

What happens if a run is interrupted, or a webhook fires twice?

Every write is idempotent. Already-scored candidates are skipped on the next run, so a duplicate webhook, a manual re-trigger, or a crash-and-restart never double-contacts a candidate or duplicates a match. A re-run safely resumes instead of starting damage.

How fast can it be adapted to a new client or database?

Nothing is hard-coded. Behavior is driven entirely by environment variables, so the same codebase redeploys to a new database, setting, or reminder timeframe in about 5 minutes, for example the matching engine on DeepSeek Flash versus Pro, or the WhatsApp reminder service deployed three times with different "hours since submission" thresholds.

Technical Writing

The documentation is part of the deliverable

Every service ships with an operational handbook: what each branch does, every environment variable, the exact eligibility rules, the log lines to grep, and a troubleshooting guide written for whoever is on call at 2am, including future me. For an automation system that runs unattended, the docs are the reliability layer.

Audience-layered

Each handbook opens with a one-table overview a manager can read, then drills into branch-level specifics a developer needs. Same document, two reading depths.

Operational, not decorative

Documented exactly which log line proves which behavior, so debugging is "grep this string," not "read the source."

Failure-first

A troubleshooting section maps real symptoms ("candidate wasn't matched") to the precise gate that excluded them.
