Impact

What it delivers, and how it works

The whole point: manual processes that used to take weeks now finish in seconds, minutes, or at most a couple of hours. Two audiences read a system like this differently, so here is both the outcome and the workflow that produces it.

For the business

  • A single 90 to 120 minute run on Railway scans the full 40,000-candidate database and scores upward of 20,000 of them, returning a ranked, explained shortlist for a role, work that is otherwise weeks of manual screening.
  • Completed interviews are processed automatically: the transcript is fetched in full and the candidate's record and status are updated in Notion within about 60 seconds, with no recruiter touching the card.
  • Communications run at scale every day: 2,000+ WhatsApp messages and emails, with CV parsing done in under a minute.
  • The team stays informed automatically: each matching run posts Slack status updates at start, progress, and completion, and every match ships with a written reason.

For the technical team

  • AI only where it earns its keep: DeepSeek is reserved for judgment-heavy steps; PDF/text extraction, filtering, and routing run on regex and deterministic logic, so tokens are never wasted.
  • Event-driven: a workspace change fires a webhook, the right Railway service handles it, the result is written back.
  • A run makes 20,000+ LLM API calls, kept inside quota by a global semaphore, a sequential job queue, and Retry-After-aware backoff.
  • Transcript fetches poll for up to 10 minutes and refuse to write until every answer is transcribed, so scoring never runs on a partial interview.
  • Idempotent re-runs, per-candidate error isolation, and schema-drift defense keep an unattended run from corrupting data.

Architecture

How data moves through the platform

Most work is event-driven off the recruiting team's workspace: a record changes, a webhook fires, the right service handles it and writes the result back. Time-based reminder sweeps run on a cron schedule instead. No shared database to corrupt, no monolith to coordinate.

Every box is an isolated, independently deployable service. Failure in one (e.g. WhatsApp) cannot block another (e.g. matching). The workspace is the single source of truth; services are stateless and idempotent.

Project 01 · Role Matching Python · FastAPI · HTTPX · DeepSeek · Railway

Six stages from job description to ranked shortlist

The hard problem isn't calling an LLM, it's calling it responsibly on a pool of tens of thousands: cheaply, in priority order, with explainable output, and without melting API quotas. Each stage narrows the pool before the next, expensive one runs.

1

Rubric extraction LLM

A free-text job description becomes a structured rubric: hard gates, core vs. preferred requirements, search keywords, and allowed locations. This turns fuzzy hiring intent into something the rest of the pipeline can filter and score against deterministically.

2

Keyword pool reduction cheap filter

A fast, local keyword-overlap pass against candidate metadata drops candidates with no plausible signal before any paid LLM call. This is the single biggest cost-control lever in the system.

3

Hard eligibility filters rules

Non-negotiable gates, language, location, minimum screening score, applied as rules. Missing or unexpected data fields are tolerated and logged, never crashed on (schema-drift defense).

4

Priority bucketing & ordering heuristics

Survivors are bucketed by business priority (referrals first, then priority backgrounds) and ordered by the best available baseline score. Higher-value candidates are scored first, so a capped run still spends budget on the right people.

5

LLM scoring semaphore-capped · 7 retries

Each candidate gets a 0-100 score and a written rationale from DeepSeek, under a global concurrency cap. Up to 20,000+ scoring calls run per broad role. Transport errors, empty completions, and invalid JSON are retried; a single candidate's failure is isolated and skipped rather than killing the run.

6

Qualification & backfill targeting

Candidates above threshold are linked first. If the qualified pool is short of the run's target, the engine backfills with the highest-scoring below-threshold candidates, so the team always gets a usable shortlist, and the fallback is logged transparently.

Reliability & error handling (how it survives 20,000+ calls a run)

Drawn from the actual service. This is what separates a production engine from a loop that calls an API.

How are DeepSeek failures and bad responses handled?

Scoring calls go out over raw HTTPX (chosen over the SDK to control the auth headers and debug 401s directly), each with a 90-second timeout and up to 7 retries using exponential backoff plus jitter, capped at 60s per wait. The handling is status-aware: a 429 honours the Retry-After header when present and otherwise backs off; 5xx server errors retry; any other 4xx fails fast, because retrying a malformed request just wastes time. Timeouts, connection/read errors, empty completions, a missing content field, and invalid JSON are all retried as one RetryableLLMOutputError that logs the finish reason, the message keys, and a content preview, so a bad batch is diagnosable from the logs.

How are Notion's rate limits and pagination failures handled?

Every Notion call runs through a retry wrapper that allows up to 10 attempts inside a 180-second window, with jittered backoff capped at 45s and a longer base delay for 503 availability errors than for ordinary transient ones; it honours Retry-After and uses a 120s client timeout. An expired/invalid pagination cursor is treated as fatal in place, you can't retry it, so during a full crawl it triggers a 5-minute cool-down and a single clean restart. A truncated ~2 MB JSON response is deliberately not retried, since the same page size would just fail again; the fix is a smaller page size, and the log says so.

How is concurrency controlled across 20,000+ scoring calls?

A single global async semaphore (default 100) caps total DeepSeek load across every job running at once. Scoring uses a prefetch sliding window: the next candidates start as earlier ones finish, so throughput stays high without ever exceeding the cap, and in-flight tasks are cancelled the instant the run's target is met. Writes back to existing Notion rows are spaced by a configurable delay (default 30s) so the write path never trips the rate limiter.

Why a job queue with a cooldown instead of running jobs in parallel?

Jobs run through a single sequential queue, one at a time, with a deliberate 5-minute cooldown between jobs (60s if a job errored) to let Notion's rate limits fully recover before the next heavy crawl. The queue tracks each job's position, and a candidate cache (TTL, default 15 min, keyed by the active filter) is only marked fresh after a job fully succeeds, so a failed run never serves a half-built pool to the next one.

What guarantees a run can't corrupt data or double-link?

Writes are idempotent: already-scored rows are excluded from future runs, so a duplicate webhook, a manual re-trigger, or a crash-and-restart can't double-link a candidate. Each candidate is scored in its own task with per-candidate error isolation, and the run only fails if zero usable scores come back, rather than letting one bad call abort 20,000 good ones. On success or failure it posts a Slack message with a direct Notion link.

Project 02 · Candidate Flow Node · Express · 14 services · cron + webhooks · Railway

14 services for everything around the match

One codebase, branch-per-automation, each deployed as its own service: outreach, interview ingestion and scoring, and CV parsing. Some are webhook-triggered (instant), some run on a cron schedule (reminder sweeps). Here is the lifecycle of a candidate.

A

On apply webhook · pdf-parse + regex

A webhook fires the CV service the moment an application lands. The CV text is extracted deterministically with pdf-parse and regex normalization, then DeepSeek is used only to structure and score it (for example a 0-5 score for portfolio links like GitHub or LinkedIn). The structured profile is written back to Notion in under a minute.

B

Outreach Postmark + Twilio · webhook + cron

Invites and reminders go out over email (Postmark) and WhatsApp (Twilio), both instantly on a webhook and on cron sweeps that check who is due against a configurable "hours since submission" threshold. The same WhatsApp service ships five message templates (confirmation, invite, reminder, final reminder, completed).

C

Interview completed Hireflix/Fireflies · 10-min poll

The interview webhook returns 200 immediately and processes in the background. It polls the transcript for up to 10 minutes at 60-second intervals and only writes once every answer is transcribed, then updates the candidate's record and flips their status automatically, usually within ~60 seconds.

D

Transcript scoring DeepSeek · retried

Once the full transcript is in, a DeepSeek service scores it against the role (fit, consistency, communication clarity) and writes a structured score plus rationale back to Notion, with retries on transport, empty-response, and JSON errors.

Reliability & error handling (2,000+ messages a day, unattended)

Drawn from the actual services. Sending at volume to third parties is mostly an exercise in not getting throttled or double-sending.

How does email stay inside Postmark's limits and never double-send?

The email service enforces hourly and daily send caps using a sliding window of send timestamps, and throws a cap error rather than overshoot. Requests retry on 408 / 429 / 5xx with exponential backoff (default 3 attempts), and on a transient error the sequential send queue cools down for 5 minutes before continuing. Sends are spaced ~1 second apart, and a WhatsApp follow-up only fires ~2 minutes after a confirmed successful email, so the same candidate is never hit twice at once.

How does the transcript fetch guarantee a complete interview?

It counts how many answers actually have transcription text and compares that to the question count, refusing to proceed until every answer is transcribed, with a marker-based parser as a fallback when the provider returns a single merged block. It polls for up to 10 minutes at 60-second intervals; if that window passes it degrades gracefully to writing whatever partial transcript exists, or just the URL and status, rather than hanging. Calls to the interview API are throttled to roughly 3 requests per second, and a candidate's email must match exactly after trimming whitespace before any row is touched.

How do WhatsApp reminders avoid spamming candidates?

Reminders are driven by cron, not a tight loop. Each sweep selects only candidates past a configurable "hours since" threshold (default 24h) and advances them through a staged template sequence (invite → reminder → final reminder), writing a "reminder sent" timestamp so the next sweep skips anyone already nudged. Deploying the same service three times with different thresholds produces a staggered cadence without any code change.

How are Notion calls and transient failures handled here?

Every Notion request goes through a wrapper that retries 429 and 5xx (and 408/409 where relevant) up to a configurable max (default 3) with exponential backoff, wrapped in an AbortController timeout (default 15s) that is itself retried. An invalid pagination cursor gets a cool-down and a bounded retry. A processing guard runs both as a server-side query filter and a local re-check that tolerates missing or wrong-typed properties, so a schema edit downgrades to a logged warning instead of a crash.

What keeps the services alive and clean under Railway?

Each service responds 200 to its webhook first and does the real work in the background, so the provider never times out waiting. There are global uncaughtException and unhandledRejection handlers, a graceful SIGTERM shutdown, and cached data-source ids and schemas to cut latency. The Notion API version is pinned so an upstream change can't silently alter behaviour.

Configurable by Design

One codebase, many deployments, zero forks

Nothing is hard-coded per client. Behavior is driven entirely by environment variables, so the same service is deployed as many times as needed, each instance tuned to a different goal, database, or client, by changing config and redeploying. The code never forks.

Same engine, Flash or Pro

The matching engine runs as two deployments from one codebase: one on DeepSeek Flash for speed and low cost, one on DeepSeek Pro for tougher roles. A single env var picks the model.

One reminder, deployed 3×

The WhatsApp reminder service is deployed three times, each with a different "hours since submission" threshold, so candidates get a staggered sequence of nudges, all from the same code.

Same code, different database

Each branch points at a different Notion database in the same workspace through env vars. New database, new deployment, no code change.

⚙︎ The payoff: when a client wants the same automation pointed at a different database, tuned to a different setting, or run on a different reminder timeframe, it redeploys to a new space in about 5 minutes. New environment variables and a redeploy, never new code.

Technical Writing

The documentation is part of the deliverable

Every service ships with an operational handbook: what each branch does, every environment variable, the exact eligibility rules, the log lines to grep, and a troubleshooting guide written for whoever is on call at 2am, including future me. For an automation system that runs unattended, the docs are the reliability layer.

Audience-layered

Each handbook opens with a one-table overview a manager can read, then drills into branch-level specifics a developer needs. Same document, two reading depths.

Operational, not decorative

Documented exactly which log line proves which behavior, so debugging is "grep this string," not "read the source."

Failure-first

A troubleshooting section maps real symptoms ("candidate wasn't matched") to the precise gate that excluded them.
